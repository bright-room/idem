# Implementation Plan Skill

GitHub Issue を参照し、コードベースを調査した上で実装プランを作成する。

## 前提条件

- 引数 `$ARGUMENTS` に Issue 番号が指定されていること（必須）
- `gh` CLI が認証済みであること

## 手順

### 1. Issue の取得

```bash
ISSUE_NUMBER="$ARGUMENTS"
```

Issue 番号が指定されていない場合はエラーメッセージを出力して終了すること。

```bash
gh issue view ${ISSUE_NUMBER}
```

Issue のタイトル、本文、ラベル、コメントを取得し、要件を正確に理解する。

### 2. コードベースの調査

Issue の要件に基づいて、関連するコードを徹底的に調査する。

- **既存の実装パターン**: 同様の機能がどのように実装されているかを確認する
- **影響範囲**: 変更が影響するモジュール、クラス、テストを特定する
- **拡張ポイント**: Auto-configuration、SPI、設定プロパティなど、活用すべき拡張ポイントを確認する
- **テスト構成**: 既存のテスト構成（ユニットテスト、統合テスト）を確認する

### 3. 実装プランの作成

以下の観点で実装プランを設計する:

- **実現可能性**: 要件が現在のアーキテクチャで実現可能かを検証する
- **設計判断**: 複数のアプローチがある場合は、各アプローチのトレードオフを分析する
- **段階的な実装**: 変更を論理的なステップに分割し、各ステップの依存関係を明確にする
- **テスト戦略**: 各変更に対するテスト方針を策定する

### 4. 実装プランの出力

実行環境に応じて出力先を切り替える。環境変数 `CI` の有無で判定すること。

#### ローカル実行時 (`CI` が未設定)

実装プランを `.claude/outputs/plans/` ディレクトリにファイルとして出力する。

- ディレクトリが存在しない場合は作成すること
- ファイル名: `PLAN-<Issue番号>-<Issueタイトルをケバブケースに変換>.md`
  - 例: Issue #42 "Add WebFlux support" → `PLAN-42-add-webflux-support.md`
  - タイトルに含まれる特殊文字は除去し、スペースをハイフンに変換する

#### GitHub Actions 実行時 (`CI=true`)

作成した実装プランを GitHub Issue にコメントとして投稿する。

```bash
gh issue comment ${ISSUE_NUMBER} --body "$(cat <<'EOF'
<実装プランのMarkdown>
EOF
)"
```

## 注意事項

- Issue の要件を正確に理解し、過不足のないプランを作成すること
- 推測ではなく、実際にコードを読んで確認した事実に基づいてプランを作成すること
- 実装ステップは具体的に記述し、各ステップで変更するファイルパスを明示すること
- 既存のアーキテクチャパターンや命名規則に従った設計にすること
- 不明点や判断が必要な事項は「確認事項」セクションに明記すること
- 変更のない「設計判断」「確認事項」「今後の展望」セクションは省略してよい
- 「今後の展望」は Issue のスコープ外の関連拡張を記載する。実装プラン策定時にコードベースを調査する中で見つけた改善機会や、自然な次のステップとなる機能拡張を挙げること
- 実装に伴うドキュメント（`README.md`、`CLAUDE.md`）の更新を必ず「ドキュメント更新」セクションに記載すること。新機能・設定変更・アーキテクチャ変更がある場合、対応するドキュメントの更新箇所を具体的に特定すること
- 設定プロパティを追加・変更する場合は `core/src/main/resources/META-INF/additional-spring-configuration-metadata.json` への追加も必ず「ドキュメント更新」セクションに記載すること
